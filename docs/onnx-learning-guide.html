A from-scratch guide, written from inside a real competition

ONNX, taught by golfing 400 tiny neural networks

This guide assumes you know Python and NumPy and nothing about ONNX. Every example is a real graph pulled from this repository — built while competing in NeuroGolf 2026, a Kaggle competition where the only thing that matters is how small a correct ONNX network can be. That constraint turns out to be a remarkably good way to actually learn what ONNX is, because you can't hide behind a framework doing the work for you — every byte is visible and every byte costs points.

What is ONNX?

Start with the problem it solves, not the acronym.

Say you train a model in PyTorch. Now you want to run it somewhere PyTorch isn't installed — a phone, a browser, a tiny embedded chip, or (as here) a Kaggle grading sandbox that only has a bare onnxruntime package. You need a way to describe "here is a sequence of math operations, and here are the numbers baked into them" that doesn't depend on the framework that built it.

ONNX (Open Neural Network Exchange) is exactly that: a file format for describing a computation as a graph, plus a spec for what each operation in that graph is allowed to mean. A .onnx file is a serialized Protocol Buffer message — a compact binary tree of nested records — that any ONNX-compatible runtime can load and execute without ever seeing PyTorch, TensorFlow, or whatever built it.

🔑 the mental model that matters most

An ONNX file is not code you can inspect line-by-line the way you'd read Python. It's a graph: a set of named tensors (data) connected by named operations (nodes). There is no control flow in the normal sense — no if, no for loop over a variable number of iterations (a few exceptions exist as ops, but this competition explicitly bans them). Every path through the graph runs, every time. If you want conditional behavior, you build it out of Where (a vectorized select) instead of a branch.

This matters enormously for how you think about problems. In Python you'd write if x > 0: y = a else: y = b. In ONNX you compute both a and b always (they're cheap, so this is fine) and then use Where(x > 0, a, b) to pick per-element. Every "decision" in an ONNX graph is a data operation, not a jump.

Why this competition uses ONNX so unusually

Normally ONNX graphs come from a trained model — the weights are learned, the graph shape mirrors whatever architecture the framework exported. Here, there's no training at all. Every one of the 400 tasks is a small, deterministic, purely symbolic transformation (rotate this grid, recolor these cells, extract this sub-region) that we hand-derive the rule for, then hand-build the smallest ONNX graph that implements exactly that rule — perfectly, with zero tolerance for error. It's less "machine learning" and more "using the ONNX instruction set to write a tiny, extremely literal program," which turns out to be one of the most direct ways to actually learn what the format's operators can and can't do.

Anatomy of a graph

Four nested containers, and everything else is detail.

ContainerHoldsAnalogy
ModelProtoOne GraphProto, plus the opset version(s) and IR versionThe whole .onnx file
GraphProtoA list of nodes, plus named inputs, outputs, and initializersA function body
NodeProtoOne operation: an op type, named input tensors, named output tensors, attributesOne line of code, e.g. y = np.transpose(x)
TensorProtoActual numeric data with a shape and dtype — used for constants ("initializers")A NumPy array literal baked into the file

A node's inputs and outputs are just strings — names. The graph is wired together purely by name-matching: if node A produces an output called "cols" and node B lists "cols" as an input, data flows from A to B. There's no other notion of "connection." This is why every task's file always has an input tensor named exactly "input" and an output tensor named exactly "output" — the grader looks those names up directly.

The simplest possible real example

Task 241 in this competition is "transpose the grid" — swap rows and columns. Because Transpose is a single op with a fixed permutation and no learned data at all, the entire solution is one node, zero initializers:

input
shape [1,10,30,30]
Transpose
perm=[0,1,3,2]
output
shape [1,10,30,30]

That's a complete, valid, submittable ONNX model. It costs exactly 0 in this competition's scoring (no initializer data, no intermediate tensor between input and output) — the theoretical minimum, worth the maximum 25 points. Most real tasks need more machinery, but this is the floor everything else is measured against.

What that graph looks like as an actual file, in Python

# scripts use onnx.helper to build NodeProto/GraphProto/ModelProto by hand
import onnx
from onnx import helper, TensorProto

node = helper.make_node(
    "Transpose",
    inputs=["input"],
    outputs=["output"],
    perm=[0, 1, 3, 2],       # swap the last two axes (H and W)
)

graph = helper.make_graph(
    [node],
    "task241",
    [helper.make_tensor_value_info("input",  TensorProto.FLOAT, [1, 10, 30, 30])],
    [helper.make_tensor_value_info("output", TensorProto.FLOAT, [1, 10, 30, 30])],
)

model = helper.make_model(graph, opset_imports=[helper.make_opsetid("", 18)])
onnx.save(model, "task241.onnx")

Real, runnable code — this is (almost) verbatim what produced the actual submitted file.

Running a graph, and reading its cost

Two things you'll do constantly: execute a model, and measure it.

Execution with onnxruntime

import onnxruntime, numpy as np

session = onnxruntime.InferenceSession("task241.onnx")
x = np.zeros((1, 10, 30, 30), dtype=np.float32)   # one-hot encoded grid
x[0, 5, 0, 1] = 1.0                                # color 5 at row 0, col 1

result = session.run(["output"], {"input": x})[0]
# result[0, 5, 1, 0] == 1.0  -- row/col swapped, exactly what Transpose promised

How this competition's cost is actually computed

This is the part that makes the whole exercise interesting: the score isn't about accuracy (a wrong answer scores zero, full stop) — it's about size. Specifically: 

cost = memory_bytes(every tensor except the ones literally named "input"/"output")
     + params_element_count(every initializer, counted in elements, not bytes)

points = max(1, 25 - ln(cost))

Two consequences shape almost every decision in this guide's later sections:

  • The input and output tensors are free. No matter how big they are, they never count. Everything between them does.
  • The score is log-scaled. Going from cost 1,000 to cost 500 is worth the same points as going from cost 100,000 to cost 50,000 (both are ln(2) ≈ 0.69 points). This means a small task is never "not worth optimizing" — the percentage cut is what matters, not the absolute byte count.

💡 the one measurement trick worth knowing

You get the byte cost of every intermediate tensor from an onnxruntime execution profile (run with enable_profiling=True, then read the shapes it recorded), and the parameter count directly from the model's initializer list. This project's verify_one.py script does exactly that on every candidate before it's ever considered — see the toolkit section.

⚠️ where the analogy breaks

NumPy lets you write dynamic-shape code freely (a[:n] where n is a runtime value, arbitrary boolean masking, ragged results). ONNX graphs in this competition must have fully static output shapes — the grader rejects anything else. So "find where the content is and crop to it" — trivial in NumPy with boolean indexing — becomes real engineering: compute the bounding box with reductions, then build a fixed-size output and use Where or padding to fill in the parts that don't apply. This gap is where most of the actual difficulty in this competition lives.

Math → ONNX

The operators read almost identically to how you'd write the math on paper, once you know the vocabulary.

Linear algebra

Matrix-vector and matrix-matrix products are MatMul, following NumPy's broadcasting-batched-matmul semantics exactly: C[...,i,j] = Σₖ A[...,i,k] · B[...,k,j]. If you also need a bias term and scalar coefficients — the full affine form y = αAB + βC — that's Gemm, restricted to strictly 2-D inputs.

Einstein summation — the single most useful op in this whole project

If you know the einsum notation from NumPy or PyTorch, ONNX's Einsum is the exact same string syntax. Given an equation like "nchw,co->nohw", it means: for every output index combination (n,o,h,w), sum over every index that appears on the left but not the right (here, c) of the product of the matching input elements. It subsumes matmul, batched matmul, outer product, trace, transpose, and arbitrary tensor contractions — all as one op, one graph node, and (critically for this competition) it lets an ONNX-optimized runtime plan the whole contraction without materializing intermediate tensors between separate ops. Full deep dive below.

Modular / bitwise arithmetic

A surprisingly large fraction of "compact lookup table" tricks in this competition come down to expressing "pick one of N precomputed values based on a small integer key" using bit manipulation instead of a full Gather over a big table. If you pack several small values into one integer (e.g. four 8-bit numbers into one 32-bit word), you can pull one back out with y = (packed >> (8·i)) & 0xFF — which in ONNX is exactly BitShift then BitwiseAnd. This is real arithmetic, directly transcribed; nothing ONNX-specific about the math, just about which op spells it.

Piecewise / conditional functions

A math definition like 

f(x) = { a(x)  if  x > 0
       { b(x)  otherwise

becomes, verbatim: 

cond = Greater(x, zero)
out  = Where(cond, a_x, b_x)   # both a_x and b_x are computed unconditionally first

This is the single biggest mental shift coming from imperative code: you never save work by skipping a branch — you compute every branch and select afterward. For this competition's grid sizes that's always cheap enough not to matter, but it's worth internalizing because it changes how you'd even think about "efficiency" compared to normal programming.

The cost model, and why it changes every instinct

Everything in the next three sections follows from one formula. 

cost  = Σ (bytes of every tensor except "input"/"output")
      + Σ (element count of every initializer)

points = max(1, 25 − ln(cost))

✓ what this rewards

Fusing steps into fewer, larger-throughput ops that never materialize an intermediate. Using the smallest dtype that's still exactly correct. Reusing one small constant instead of two "different-looking" ones that turn out to be identical. Exploiting attribute-only variants of ops (see Opset archaeology) so a parameter never has to exist as a stored tensor at all.

✗ what it punishes, counterintuitively

Fewer nodes is not the goal — fewer bytes is. Splitting one op into two "smaller-looking" ops can massively increase cost if it introduces a big intermediate tensor between them. We measured this directly: replacing a single 5-parameter MaxRoiPool with a "cheaper-looking" 0-parameter Slice+Upsample pair raised the real cost from 5 to 36,000 — the intermediate tensor between the two ops got charged in full.

Params are counted by element, memory is counted by byte — this asymmetry is exploitable

A parameter (initializer) tensor's cost is its element count, full stop — dtype doesn't matter. A ten-value lookup table costs 10 whether it's stored as int64 or int8. But an intermediate tensor's cost is bytes = dtype_size × element_count — dtype matters enormously there. This means: pick the cheapest safe dtype specifically for large computed tensors that live between ops; don't bother for small constant tables, it won't move the needle.

Worked example: quantization cuts a real tensor 4×

Conv on float32 input produces float32 output — 4 bytes per element, even if every actual value is a small non-negative integer (which is common here: we're literally counting or matching colors, 0–9). Swap it for QLinearConv, ONNX's quantized convolution, and the same arithmetic runs on uint8 — 1 byte per element, with zero precision loss as long as values never leave the uint8 range (they don't, for a color-counting task). That's a flat 4× cut on whatever intermediate tensor the Conv produces, for free. 

# float32 Conv: 4 bytes/element on the output tensor
y = Conv(x_float32, weight_float32)        # charged at 4×H×W×C bytes

# QLinearConv: same math, uint8 output -- 1 byte/element
y = QLinearConv(x_u8, x_scale, x_zp,
                w_u8, w_scale, w_zp,
                y_scale, y_zp)             # charged at 1×H×W×C bytes -- 4x cheaper

Every op family we actually used, and why

Grouped by job, not alphabetically — this is the order you'd reach for them in.

Reshaping & indexing — moving data around without changing values

ReshapeTransposeSlice GatherGatherNDGatherElements ConcatSplitSqueeze UnsqueezeScatterElementsScatterND

These cost nothing computationally (no arithmetic) but their output tensor still counts toward cost if it's not the final "output". The rule of thumb that came out of a hundred small trials: the fewer distinct intermediate tensors your reshaping chain produces, the better — often worth accepting a slightly "uglier" single-op formulation over a readable multi-step one.

Arithmetic & logic — the actual computation

AddSubMulDiv ModNegSign BitShiftBitwiseAnd/Or/Xor/Not EqualGreater(OrEqual)Less(OrEqual) And/Or/Not/XorWhere

Straightforward elementwise math, all broadcasting like NumPy. Where deserves special mention as covered above — it's the entire conditional-logic toolkit for a format with no real branching.

Reduction — collapsing an axis to a summary

ReduceSumReduceMaxReduceMin ReduceL2ArgMaxArgMin GlobalMaxPoolGlobalAveragePool

Used constantly to answer "how big is the actual content" or "which color appears most" — the kind of dynamic-size-detection NumPy would do with boolean masking, but expressed as a pure reduction so the output shape stays static.

Convolution family — repurposed as a spatial pattern-matcher, not a learned filter

ConvConvTransposeConvInteger QLinearConvMaxPoolMaxUnpool MaxRoiPoolRoiAlignGridSample

In normal deep learning, a Conv's weights are learned. Here, we hand-derive an exact small kernel that implements a specific rule — "detect this 3×3 pattern," "count neighbors of this color," "crop this region and upscale it 2×" (MaxRoiPool, which takes a fixed region-of-interest tensor and pools+resizes it to a target size in one op — the cheapest way found to do crop-and-scale together).

The permutation trick — small tables via Gather, not full one-hot recoloring

Several tasks are "remap color A to color B, everything else stays the same." The cheapest correct implementation found: a single Gather(input, index_array, axis=1) where index_array is a 10-element list saying, for each output channel, which input channel to copy from (identity for unchanged colors, swapped for the two that change). Ten elements, one op, no intermediate — beats every decomposition that tried to isolate just the changed channels (see the 36,000-vs-5 story for why "isolate the small part" backfires here).

Deep dive: the single-Einsum collapse

The highest-value pattern found in this entire competition, worth 0.5–3 points per task, repeatedly.

A lot of tasks decompose naturally into a pipeline: extract a feature, multiply it by a lookup, sum over an axis, multiply by another lookup, place the result. Written as separate ONNX ops, each step produces a real intermediate tensor that gets charged. But Einsum can express an entire multilinear pipeline — many inputs, many contracted indices — as a single subscript equation, and a good runtime plans the whole contraction without ever materializing the in-between steps as separately-costed tensors.

input
const A
const B
const C
Einsum
one contraction,
every index at once
output

One node between input and output. No intermediate tensor exists to be charged, regardless of how many logical "steps" the equation encodes.

A real equation from one of this project's tasks (edited for length, same structure):

equation = "ha,ad,bd,be,ef,hf,hg,gi,ri,rj,jk,hk,rl,lm,op,rp,nzxy,...,nchs->ncrs"
output = Einsum(Q, A, C, C, D, Q, Q, M, Q, Q, M, Q, Q, A, D, Q, input, weights..., equation=equation)

Reading an equation like this: each comma-separated group before the arrow names the axes of one input tensor by letter; any letter that doesn't appear after the arrow gets summed over. It looks intimidating with 30+ operands, but it's built the same way you'd build any einsum — one small, understandable multiplication at a time on paper — and then simply concatenated into one string once you've confirmed it's correct piece by piece.

✓ when to reach for this

Whenever the transformation is a chain of steps that are each linear or bilinear in their inputs (weighted sums, outer products, masked selections expressed as multiplication by an indicator). If a step genuinely needs a non-linear function (comparison, rounding, a reduction that isn't a sum), it can't join the Einsum and has to stay a separate op — that's fine, just keep everything else fused around it.

Quantization ops, explained without the ML-deployment jargon

"Quantization" usually means shrinking a trained model for deployment. Here it just means picking the right integer type for arithmetic that was always going to be small integers.

⚠️ a landmine specific to this competition's pinned runtime

The real grader runs a specific pinned onnxruntime version that turned out to have no kernel implementation for TopK on uint8/int8 inputs — even though the ONNX spec's own type-constraint table allows it. It passes silently on a newer local onnxruntime and only fails on the exact pinned version. The fix was a Cast to int32 immediately before TopK and back after — but that cast can itself cost more than the quantization saved, so it always needs to be measured, not assumed. The lesson generalizes: a spec allowing a dtype combination doesn't guarantee every real runtime implements it — always test against the exact pinned version you'll be graded on, not just the newest one you have installed.

War story: a real memory-safety bug, hiding in plain sight

The single most consequential bug found in this whole project — and it wasn't in our code at all.

Conv and ConvTranspose both accept an optional bias tensor: one value added per output channel. The ONNX spec says its length should equal out_channels. Nothing in onnx.checker.check_model, strict shape inference, or a normal execution actually enforces that length match — a bias tensor with fewer elements than channels loads fine, runs fine, and produces an answer.

The catch: for the channels beyond the bias array's actual length, the runtime kernel reads whatever bytes happen to sit in memory right after the array — undefined behavior, not a bounds-checked error. On a freshly-started process, that memory is usually zeroed, so the model looks completely correct in every local test. But a grading environment that reuses the same process across many submissions can have anything sitting in that memory — leftover data from a previous, unrelated model's execution.

bias: [b₀, b₁, ..., b₈]
9 values stored
but out_channels = 10, so channel 9 reads →
??? whatever's
next in memory

This explained every "identical zip scores differently between submissions" mystery this project hit during the day — including two separate ~18-point unexplained score swings that had, at the time, been (wrongly) written off as random grading noise. It was independently confirmed by several other competitors on the forum, and the competition host acknowledged it as a real ONNX Runtime bug. The fix is mechanical once you know to look for it: 

def check_conv_bias(model):
    """Conv/ConvTranspose nodes whose bias length != out_channels."""
    init = {t.name: t for t in model.graph.initializer}
    bad = []
    for n in model.graph.node:
        if n.op_type not in ("Conv", "ConvTranspose") or len(n.input) < 3:
            continue                       # bias input is optional
        w, b = n.input[1], n.input[2]
        group  = next((a.i for a in n.attribute if a.name == "group"), 1)
        dims   = list(init[w].dims)
        out_ch = dims[0] if n.op_type == "Conv" else dims[1] * group
        blen   = init[b].dims[0] if init[b].dims else 0
        if blen != out_ch:
            bad.append((n.name, blen, out_ch))
    return bad   # empty = clean; fix is to zero-extend bias to out_ch elements

the takeaway that generalizes past ONNX entirely

"It runs without error and produces plausible output" is not proof of correctness for anything that touches raw memory across an optional/variable-length field. Static structural checks (does this length match that length?) catch a whole category of bug that no amount of running the file locally ever will, because a fresh process's zeroed memory hides the exact symptom you're trying to find.

War story: opset archaeology

ONNX operators change shape across spec versions — sometimes an older version is strictly cheaper.

Every ONNX model declares an opset version per domain, and an operator's accepted inputs/attributes can differ across versions of the same op name. Slice took starts/ends/axes as plain integer attributes at opset 1–9; from opset 10 onward, the exact same information has to arrive as separate tensor inputs, which then count toward cost. Upsample (deprecated in favor of Resize at opset 10) similarly took its scales as a float-list attribute at opset 7–9. 

from onnx import defs
for v in (1, 7, 9):
    s = defs.get_schema("Upsample", v)
    print(v, [i.name for i in s.inputs], list(s.attributes))

# 1  ['X']          ['mode', 'width_scale', 'height_scale']
# 7  ['X']          ['scales', 'mode']            <- single input, scales is an attribute!
# 9  ['X', 'scales'] ['mode']                       <- scales became a required tensor input

Nothing stops a model file from declaring an old opset version deliberately to use the cheaper attribute-based form of an op — the ONNX runtime executing it doesn't care that a newer, different-shaped version of the same op name also exists. This is worth checking with onnx.defs.get_schema(op_name, version) any time an op's tensor inputs feel like they're carrying information that's actually static and known at graph-build time.

⚠️ but verify the end-to-end cost, not just the params count

This is exactly the trap from the cost-model section: an attribute-only op is only a win if it doesn't force you into a multi-node decomposition with a big intermediate. We tested replacing a 5-parameter, single-node MaxRoiPool (crop+scale together) with a 0-parameter Slice(opset 1, attributes) + Upsample(opset 7, attributes) pair — technically zero stored parameters, but the tensor flowing between the two nodes cost 36,000. The "free" trick made things 7,000× worse. Always measure through the real scorer before trusting a parameter-count argument in isolation.

War story: a clean local test is necessary, not sufficient

The gap between "passes every example I have" and "is actually correct" is where this project lost — and later recovered — the most points.

Every task ships with a few hundred cached example input/output pairs. A candidate passing all of them feels like proof. It isn't — it's proof the candidate matches the specific examples someone happened to generate before. Two categories of failure showed up repeatedly:

1. The candidate overfits an assumption that happens to hold across the cached sample

A rule like "the interesting region is always in the top-left 15×15" might be true for every one of 266 cached examples and false for the 267th one nobody's tested yet. The fix: this competition's official task generator (the exact program that produced the original examples) is publicly available — running it fresh, at volume, catches assumptions that happened to survive the fixed cached sample purely by luck. One task looked completely solved at 266/266 locally and turned out to fail 32 times out of 400 freshly-generated examples.

2. The test script itself doesn't replicate the real grader's own quirks

The official conversion function silently skips (never scores) any example whose grid is larger than 30×30. A hand-rolled comparison script that doesn't replicate that exact skip produced a false "failure" rate of 74% on one task — every one of those "failures" was actually an oversized grid the real grader would never have tested in the first place. The lesson cuts both ways: don't trust a clean pass (point 1), but also don't panic over a dirty one until you've confirmed your test itself matches the real evaluation exactly.

the general principle

Whenever "correct" is defined by an external grader you don't fully control, the only trustworthy test is one that reproduces that grader's exact code path — not a plausible re-implementation of it. Import the real conversion/scoring functions directly rather than rewriting them from a description, every time it's possible.

War story: telling a real algorithm from a hash-memorizer

Some "impossibly cheap" candidates aren't clever — they've just memorized the specific examples in front of them.

A model that computes a hash/checksum of the input grid, then Gathers a pre-baked answer from a lookup table sized to match the local example count, will pass every cached test perfectly at very low cost — and score exactly zero on any input it hasn't seen, because it never learned the actual rule. This is genuinely hard to spot from the graph alone, since BitShift + BitwiseAnd/ Gather combinations are also the normal, legitimate way to build a compact packed lookup for a real structural classifier (see the permutation trick, above).

The screening process that worked reliably:

  1. Any model combining bit-manipulation ops with Gather gets flagged for a closer look, no exceptions.
  2. Check every initializer's element count against the local example count. A table with ~265–270 entries — suspiciously close to "one entry per cached example" — is a strong memorization signal. A table with 20–150 entries, clearly smaller than the example count, is more likely a genuine small structural classifier (grid size, shape count, symmetry class).
  3. The decisive test either way: run it against a batch of freshly generated, never-cached examples. A real algorithm keeps working. A memorizer fails immediately and completely, because it's never seen these specific hashes before.

✓ a real example that passed this screen

One candidate used exactly this op combination with a 38-entry packed table, on a task with ~266 cached examples. 38 ≪ 266 ruled out per-example memorization; running it against 400 freshly-generated examples it had never seen (0 failures) confirmed it was a genuine structural classifier keyed on a small number of grid categories, not a hash lookup. It was merged with confidence.

Your toolkit, if you want to try this yourself

The five-question checklist this project settled on before merging anything

  1. Does it pass onnx.checker.check_model(full_check=True), and run cleanly in both onnxruntime versions?
  2. Does the graph use any op combination that's a known landmine (BitShift+Gather without a size check; TopK on a narrow int type; Conv/ConvTranspose bias length not checked against out_channels)?
  3. Does it pass on freshly generated examples, not just the cached local set?
  4. Is the gain worth double-checking with an isolated single-task real submission, given everything above passed?
  5. If a batch submission doesn't score what the sum of individually-verified pieces predicted — is the bug really in the files, or in how the test batch was assembled?

Glossary

OpsetA versioned snapshot of "what each op means and accepts." A model declares which opset it targets; the same op name can behave differently across opsets.
InitializerA named constant tensor baked into the graph — weights, lookup tables, fixed indices. Counted by element toward this competition's cost.
NodeOne operation in the graph: an op type plus named input/output tensors and attributes.
AttributeA value fixed at graph-build time and baked directly into the node (not a separate tensor) — free in this competition's cost model, unlike a tensor input.
Value infoDeclared shape/dtype metadata for an intermediate tensor. Sometimes needs to be explicitly written (not just inferred) for the checker to accept dynamic-shaped intermediates.
Sanitize (this project's term)Rename every node/tensor to a safe canonical form before scoring, matching exactly what the real grader does before measuring cost — skipping this step means measuring the wrong thing.
QuantizationRepresenting values in a narrower integer type (uint8/int8) instead of float32, when the actual range of values allows it without losing precision.
Undefined behavior (UB)Code whose result the spec doesn't define — e.g. reading past the end of an array. May happen to "work" in one environment (zeroed fresh memory) and silently corrupt in another (reused process memory).
